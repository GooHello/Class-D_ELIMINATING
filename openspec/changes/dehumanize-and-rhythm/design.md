## Context

当前是一个 SCP 主题的 Match-3 游戏，使用 React+TypeScript+Vite 构建。核心循环是部署 D 级人员消除棋盘上的色块来推进收容进度。现有问题：关卡数据设计为无限循环（100+关才出结局），五色工种只有进度加成差异，邮件系统只是右侧面板内的折叠列表，档案系统完整展示姓名/背景/职业与"平庸之恶"主题冲突。

关键文件现状：
- `src/App.tsx`（~1500行）：主逻辑+渲染，包含 processMatches/handlePieceClick/所有 modal
- `src/data/missions.ts`：关卡定义（当前设计为无限扩展）
- `src/data/professions.ts`：国际化名字生成器+Map缓存
- `src/data/colorBonuses.ts`：`calculateWeightedProgress()` 加权进度计算
- `src/data/emails.ts`：邮件数据（少量占位内容）
- `src/components/PieceToken.tsx`：棋子渲染组件

## Goals / Non-Goals

**Goals:**
- 20关内完成完整叙事弧线，三关一小爆、五关一大爆
- 去人格化系统随关卡渐进退化（姓名→编号→批次号→匿名资源）
- 5色工种各有独特被动效果，增加策略深度
- 邮件系统改为全屏 Modal 弹出阅读，填充20封叙事邮件
- 收容进度用百分比展示
- 战报面板限高+滚动
- 全局语言官僚化，强化反讽

**Non-Goals:**
- 不做多语言国际化（只保留中文）
- 不做存档云同步
- 不做关卡编辑器
- 不重构 App.tsx 的组件拆分架构（保持单文件，减少风险）
- 不做音效/BGM系统

## Decisions

### 1. 去人格化分阶段系统

**选择**：基于 `currentLevel` 的4阶段常量映射，而非渐进式连续函数。

阶段定义：
| 阶段 | 关卡 | 档案展示 | 称谓 |
|------|------|----------|------|
| PHASE_HUMAN | 1-5 | 完整姓名+背景+入站原因 | "D级人员" |
| PHASE_NUMBER | 6-10 | 编号+工种代号，背景变为"[已归档]" | "编号单位" |
| PHASE_BATCH | 11-15 | 批次号（如 "LOT-2024-███"），无个人信息 | "耗材" |
| PHASE_VOID | 16-20 | hover面板显示 "[ 无可用数据 ]" 或 "资源#███" | "资源" |

**替代方案**：每关递减一个字段 → 太细碎，玩家感知不到明确的"阶段转变"冲击
**理由**：4个明确阶段让每次跳变成为一个叙事事件，配合邮件/弹窗提示玩家注意到变化

### 2. 工种被动效果架构

**选择**：在 `colorBonuses.ts` 中定义 `ColorPassive` 接口，processMatches 中根据消除颜色触发

```typescript
interface ColorPassiveResult {
  extraMoves?: number;        // 蓝色：+1步
  breachChance?: number;      // 红色：触发突破概率
  casualtyReduction?: number; // 绿色：折损率降低
  crossClear?: [number,number][]; // 橙色：十字清除坐标
  anomalyEffect?: string;    // 紫色：随机效果类型
}
```

**理由**：集中在一个函数中返回所有被动效果，processMatches 只需调用一次并应用结果，避免分散逻辑

### 3. 邮件系统 Modal

**选择**：全屏 Modal overlay，左侧邮件列表 + 右侧阅读区，类似邮箱客户端

**理由**：
- 解决当前窄面板展开后需要反复滚动的问题
- 全屏布局有足够空间展示长篇叙事邮件
- Modal 不会影响游戏区域的现有布局

### 4. 结局分支——人性指数

**选择**：隐藏的 `humanityScore` 从100开始，根据行为加减分

加分行为：犹豫操作（+2）、阅读特定邮件（+1-3）、伦理审查选择"人性"选项（+5）
减分行为：高效消耗无犹豫（-1/关）、忽略邮件（-1）、伦理审查选择"效率"选项（-3）

结局分支：
| 指数范围 | 结局 | 主题 |
|---------|------|------|
| 80+ | 结局A: 觉醒 | 你试图离开，却发现已经太迟——但你还是选择了记住 |
| 50-79 | 结局B: 服从 | 你成为了系统的一部分，获得晋升，档案上多了一行"表现优异" |
| 20-49 | 结局C: 替代 | 你被AI取代，自己变成了D级编号 |
| <20 | 结局D: 虚无 | 界面逐渐消失，最后只剩一个闪烁的光标 |

### 5. 进度百分比化

**选择**：UI层面转换，不改变底层数据结构。`mission.targetProgress` 仍为具体数字，显示时计算 `Math.round((totalProgress / targetProgress) * 100)` + "%"

**理由**：最小改动，不影响现有 processMatches 逻辑

### 6. 推荐工种描边

**选择**：CSS `box-shadow` 描边 + `animation: pulse`，通过 `isTargetColor` prop 已有的通道传递

**理由**：不需要修改数据流，只需修改 PieceToken 的 CSS 样式

## Risks / Trade-offs

- **[叙事内容量巨大]** → 20关数据 + 20封邮件 + 4个结局 + 去人格化文案需要大量写作。分批实现：先框架后内容。
- **[工种被动平衡性]** → 蓝色回步可能过强。限制为"3个以上匹配才触发"且每回合最多+1步。
- **[红色被动触发收容突破]** → 可能让玩家刻意避免红色。设计为"小型突破"（只重组2行而非全棋盘），概率15%。
- **[结局分数不透明]** → 玩家不知道人性指数存在。这是设计意图——但需要在邮件/伦理审查中给足够暗示让玩家感觉"选择有意义"。
- **[20关太短]** → 每关步数适当增加，确保单关体验充实（15-25步/关）。
